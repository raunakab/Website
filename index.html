UBC Bionics is a pure undergraduate, student-led engineering design team producing a robotic, prosthetic arm for patients with amputations or disabilities.
We are leading research and development into producing a low-cost arm with functionalities similar to that of a real one.
The team consists of 3 sub-teams: the mechanical engineering team, electrical engineering team, and software engineering team.
The software engineering team is further split up into 2 subteams: the embedded systems team and the analytics team.
The mechanical team is responsible for researching and developing the actual physical design of the arm.

The embedded systems team write the software controlling the entire arm; in a sense, the team is writing a "pseudo-kernel" wrapping the main OS running on the arm.
Lastly, the analytics team is writing the machine learning scripts running on the arm; these scripts are designed to take in the muscle contractions from the EMG sensors and classify them into distinct finger grips for the robotic finger digits to perform.

Raestro, v0.3.0

A Rust-flavoured library for interacting with the 6-Channel Pololu Micro-Maestro Servo-Controller-Board
https://github.com/BEARUBC/raestro
https://crates.io/crates/raestro
https://docs.rs/raestro/0.3.0/raestro/

The Raestro project was born out of necessity, as most software tools/libraries are.
In Jan 2021, the UBC Bionics Embedded Systems Team migrated over from C++ to Rust, on account of Rust's major safety guarantees.
As migration ramped up and development of the robotic prosthetic arm continued, our team needed a library to communicate with an external servo controller board called the 6-channel Micro-Maestro.
Unfortunately, given the relative 'newness' of Rust, its dependency ecosystem was still evolving and there existed no Rust library to communicate with a Maestro.
So our team decided to bite the bullet and write our own library.

The Micro-Maestro uses UART serial pins to communicate with an external microcomputer.
So using the UART GPIO pins on the Raspberry Pi 4 Model B (which was the main computer system onboard our robotic prosthesis), we connected that to the Maestro.
This is where Raestro comes in; Raestro provides a dead-simple API to send protocols implementing the standardized Pololu protocol over UART serial.
Essentially, all of that lower-level serial communications is neatly abstracted away into nice little API calls.
This project was so helpful for us, we decided to release it as UBC Bionics' very first Open-Source-Software Project, licensed under the MIT OSS License!
Up till today, Raestro is on version 0.3.0 and has over 80 downloads!

While being a wonderfully helpful OSS project and something that I'm proud of, Raestro also managed to teach me another thing:
API Design.
Yes, developing a library and constructing APIs for end-users to interact with is an art; it's not something easily tackle-able.
Our team struggled for weeks on end just trying to solidify a design; we would always try to implement a design, but would end up asking ourselves "But wait, this design doesn't make sense", or "Oh shoot, we forgot about this crucial use-case".
In the end though, it was a wonderful project to participate in developing, and it will always be marked as my first truly Open-Source-Software Project.
Please check out the above links for more information on Raestro, and leave a star on the Github page!

As a final note, with regards to the name "Raestro", it is derived as a portmanteau of "Rust" and "Maestro".

AsyncFramework, v0.1.0

A Rust-flavoured library for interacting with the 6-Channel Pololu Micro-Maestro Servo-Controller-Board
https://github.com/BEARUBC/raestro

This is another OSS project that is currently in the works, but with this one being quite a bit more complex than the previous.
The Embedded Systems Team at UBC Bionics was tasked with making the software running on board the Raspberry Pi with certain requirements for what it must do.
It needed to contain a generalized "main loop", inside of which we could run some "checks" on the status of the arm, and update an internal "state-machine" accordingly.
Now, the difference between writing a library and a framework is night and day - a library just contains some abstracted functions, whereas a framework is a generalized application into which you inject your specific functionalities.
The difference is subtle but quite significant.

AsyncFramework is a multi-threaded, asynchronous (i.e., non-blocking) software framework.
It contains the notion of "components" - essentially just structs that contain an asynchronous execution loop, an address to send messages to, and a "routine".
A routine is essentially just an array of "jobs" (almost syntactic sugar for closures/lambdas, in our case) to run in an infinite loop.
The way to work with AsyncFramework is to whip up a bunch of components, put the appropriate jobs in each of them, and pass each other's addresses around so that they can send messages to one another.

But what is the point of AsyncFramework?
Well, the biggest advantage here are its async, multi-threaded components.
Components are designed to be isolated!
The effects of any critical failures in one will be mitigated across other threads, but more importantly, after catching any fatal crashes, the component can reboot.
Components, while being isolated, can also communicate with one another using a custom data-type that the end-user designs!
That means you can decide how messages will look and how to pass them around!
And since each component contains an async loop, all message-handlers, jobs, etc., are run asynchronously.

The UBC Bionics Embedded Systems Team is hard at work releasing v0.1.0 of AsyncFramework, and we believe it has potential in many use-cases, not just bionic prostheses!
Once again, for more information, please check out the above link, and leave a star!
